# i18n Guide

Ezdoc's UI copy — view-rendered HTML, JS dialogs, and AJAX action-endpoint
response messages — is externalized into per-locale PHP array catalogs,
looked up through a small `t()` helper — server-side (PHP) and client-side
(JS), with the same key namespace on both sides. Both `id` (Bahasa
Indonesia, the default) and `en` (English) locales are populated; the
mechanism is designed so adding another locale is a matter of authoring new
catalog files, not touching view code.

## Why this pattern

Ezdoc positions itself as framework-agnostic and installable both via
Composer *and* as a dependency-free "standalone" drop-in (`bootstrap.php`,
no autoloader required). That rules out the two other common PHP i18n
choices:

- **`gettext`** — the oldest, most "standard" option, but requires the PHP
  `gettext` extension plus system locale configuration, which fights the
  standalone-drop-in story (a shared-hosting consumer may not control PHP
  extensions or server locales).
- **Symfony Translation** — mature and widely used, but a hard Composer
  dependency, same problem for standalone consumers.

Instead, this mirrors what large PHP-based CMSs already do for exactly this
situation — PHP-rendered pages with inline JS dialogs, no build step:
**plain PHP array catalogs + a small `t()` helper, with the resolved catalog
also injected into JS as one dictionary** (WordPress's `wp_localize_script`,
Drupal's `Drupal.t()`). Zero dependencies either way.

## How it works

**`Ezdoc\UI\Translator`** (`src/UI/Translator.php`) composes an
`Ezdoc\UI\Config` internally — the same relationship `Ezdoc\UI\Theme` already
has with `Config` — rather than reimplementing dot-notation lookup:

```php
$translator = \Ezdoc\UI\Translator::forView('designer', 'id');
$translator->t('actions.save', [], 'Save'); // 'Simpan'
```

`forView($view, $locale, $langDir = null)` loads, in order (later overrides
earlier on key conflict):

1. `lang/{locale}/common.php` — strings shared across views
2. `lang/{locale}/{view}.php` — strings specific to that one view

A missing file is not an error (that section is just empty); a file that
exists but returns something other than an array is logged via
`error_log()` and skipped — a broken lang file must never fatal a live
page.

**`t()` never throws.** A missing key falls back to `$default` if you
passed one, else the key itself — so a typo'd key degrades to visibly-wrong
text you'll notice in review, not a blank page or fatal error.

**Interpolation** uses single-brace `{param}` placeholders, resolved via
`strtr()`:

```php
t('confirm.delete_version', ['version' => 3], 'Delete version v{version}?');
// → 'Hapus versi v3?'
```

**Keys are NOT prefixed with the view name** (no `designer.` / `generate.`
prefix) — see [Key naming convention](#key-naming-convention) for why that
would silently break.

This is deliberately `{param}`, not `{{param}}` — the double-brace mustache
syntax is already used by ezdoc's own document-template variables
(`{{field_name}}`, `{{tabledb.ns.col}}`) and lives in a completely different
string universe (per-document author content vs. app-chrome UI copy). Don't
mix the two up.

### Wiring inside a view

Both `views/document/designer.php` and `views/document/generate.php` resolve
a `$translator` in their existing bootstrap-fallback block (same spot that
already resolves `$ctx`/`$config` for standalone-vs-Router invocation):

```php
if (!isset($translator) || !($translator instanceof \Ezdoc\UI\Translator)) {
    $translator = \Ezdoc\UI\Translator::forView('designer', (string) $config->get('app.locale', 'id'));
}
$GLOBALS['translator'] = $translator;
```

The `$GLOBALS['translator']` promotion is **required, not just tidy** —
`Ezdoc\Http\Router::renderView()` `include`s these views from inside a
*method*, so top-level variables in the included file live in method-local
scope, not global scope. A global `t()` helper reading `global $x` (or
`$GLOBALS['x']`) only sees the right value if it was explicitly mirrored
into `$GLOBALS` at assignment time. This is the exact same class of bug
fixed for `$dbFields`/`$dbTtd` in an earlier commit (search git log for
`"scope isolation"` in `generate.php`) — the fix pattern is identical here.

Each view then defines a **local, `function_exists`-guarded** `t()` helper
right next to its existing `h()` escaping helper (same convention, not
centralized in `bootstrap.php` — views are meant to be self-contained /
copy-pasteable per this library's own portability model):

```php
if (!function_exists('t')) {
    function t(string $key, array $params = [], ?string $default = null): string {
        $translator = $GLOBALS['translator'] ?? null;
        if (!($translator instanceof \Ezdoc\UI\Translator)) {
            return $default !== null ? $default : $key;
        }
        return $translator->t($key, $params, $default);
    }
}
```

### JS side

Each view injects its resolved catalog into JS as one dictionary — mirroring
whatever mechanism that file *already* uses for its own URL bag (no new
plumbing invented):

- `designer.php` round-trips through a `data-ezdoc-i18n` HTML attribute +
  `JSON.parse()` (sibling of its existing `data-ezdoc-urls` attribute).
- `generate.php` injects a literal `window.EZDOC_I18N` inline `<script>`
  const (sibling of its existing `window.EZDOC_URLS`).

Both then define a JS `t(key, params)` that walks the nested object by
splitting `key` on `.` — the same dot-path traversal algorithm as
`Config::get()`, just in JS:

```js
t('confirm.delete_template', {name: name}, 'Delete template "{name}"?');
```

## Adding/editing a string

**Keys and the `$default` fallback are always English — only the catalog
*values* in `lang/id/*.php` are Indonesian.** A key like `actions.save` is a
stable, locale-neutral identifier (the same convention gettext/Laravel/Rails
all use — English identifiers regardless of which locale is active), and the
`$default` argument is the "source string" a developer reads in code review.
Never write an Indonesian word as a key segment or as the `$default` — if
you catch yourself typing `t('toolbar.simpan', [], 'Simpan')`, that's wrong
on both counts; it should be `t('toolbar.save', [], 'Save')` with `'Simpan'`
living only in the catalog file.

**Do not prefix the key with the view name** (`designer.`/`generate.`) — see
[naming convention](#key-naming-convention) below for why.

1. Pick (or add) an **English** key under the right top-level section in
   `lang/id/{designer,generate}.php` (see [naming convention](#key-naming-convention)).
2. At the call site, wrap the literal, passing the **English** text as the
   fallback:
   - PHP: `<?= h(t('toolbar.save', [], 'Save')) ?>`
   - JS: `t('confirm.delete_template', {name: name}, 'Delete template "{name}"?')`
3. Add the key to the catalog file with its **Indonesian** value:
   `'toolbar' => ['save' => 'Simpan']`.
4. The `$default` fallback is the safety net that makes a missing/mistyped
   key fail *visibly-but-correctly* (shows correct-enough English text)
   instead of breaking — always pass one.
5. If the string is genuinely shared by both views (a plain "Save"/"Cancel"
   button, say), put it in `lang/id/common.php` under the `actions` or
   `fallback` section instead of duplicating it in both view catalogs.

## Adding a new locale

`id` (Indonesian) and `en` (English) are both populated. The `en` catalog
was generated directly from the `$default` argument already present at
every `t()` call site (not translated by hand) — that argument is the
"source string" convention this whole system is built on, so it doubles as
the English catalog's source of truth. To add another locale:

1. Create `lang/{locale}/common.php`, `lang/{locale}/designer.php`,
   `lang/{locale}/generate.php` — same key structure as `id/`/`en/`,
   translated values. Native/domain review is strongly recommended before
   shipping a new locale, especially for medical/legal document terms.
2. A consumer selects it via config: `Config::fromArray(['app.locale' => 'xx'])`
   (or however they already load their `Config`) — `Translator::forView()`
   reads `app.locale` with a default of `'id'`.
3. No view code changes needed — `t()` calls are locale-agnostic by
   construction.

**Maintenance note:** since `en`'s values were extracted from call-site
defaults, keeping them in sync going forward means either (a) editing both
the call site's `$default` and the matching `lang/en/*.php` value by hand
when English copy changes, or (b) re-running the same extraction approach
(walk all `t()` call sites, take the 3rd argument, match against the `id`
catalog's key structure) if the two catalogs drift.

## Key naming convention

Dot-path, `{section}.{subsection}.{name}` — **never prefixed with the view
name** (no `designer.toolbar.save`, just `toolbar.save`).

This isn't a style preference — `Translator::forView($view)` loads
`lang/{locale}/common.php` and merges `lang/{locale}/{view}.php`'s top-level
keys **directly, flat, with no `$view`-named wrapper**:

```php
// forView('designer', 'id') internally does roughly:
$catalog = Config::fromFile('lang/id/common.php');      // ['actions' => [...], 'fallback' => [...]]
$catalog->merge(Config::fromFile('lang/id/designer.php')->all()); // ['toolbar' => [...], 'list' => [...], ...]
// merged catalog top-level keys: actions, fallback, toolbar, list, ... — NOT 'designer'
```

So `t('designer.toolbar.save', ...)` looks up a top-level `designer` key
that doesn't exist, silently falls through to `$default` every time, and
**never renders the actual Indonesian catalog value** — a real bug that
shipped briefly during this migration before being caught (grep the file
for `t('designer.` or `t('generate.` — if that ever turns up a real hit
outside a `$config->get(...)` call, it's this bug again).

This is safe because `designer.php` and `generate.php` each build their
**own independent `Translator` instance** (`Translator::forView('designer', ...)`
vs. `Translator::forView('generate', ...)`) — neither ever loads the other's
lang file, so identical section names in both files' catalogs (e.g. both
having their own `toolbar.print`) never collide at runtime, even though
there's no view-name prefix to separate them.

Top-level sections `actions`, `fallback`, and `response` are the exception —
they live in `lang/id/common.php`, loaded by *every* Translator instance,
and are **reserved**: a view catalog must not redeclare any of them as a
top-level key, since `Config::merge()` deep-merges associative arrays (a
view file re-declaring `actions` would blend with, not replace, common.php's
`actions`, which is rarely what you want).

View-owned top-level sections currently in use (add more as needed,
following this shape — no need to keep the two views' section names in
sync with each other, they're independent catalogs):

- `lang/id/designer.php`: `toolbar`, `list`, `placeholder`, `title`,
  `editor`, `alert`, `confirm`, `sidebar`, `modal`, `field`
- `lang/id/generate.php`: `toolbar`, `title`, `placeholder`, `field`,
  `save`, `ttd`, `materai`, `validation`, `alert`, `confirm`, `toast`,
  `modal` (with `modal.doc_info`, `modal.new_version`, `modal.shortcuts`,
  `modal.verify_qr`, `modal.qr_field`, `modal.sign` sub-sections)

### Why `response.*` lives in `common.php`, not its own catalog

AJAX endpoint handlers under `actions/**/*.php` (`ezdoc_respond_success()` /
`ezdoc_respond_error()` messages, wired via `actions/_dispatcher.php`) are a
third `t()`-consuming surface, but they don't get their own
`lang/{locale}/actions.php` file. The reason: an action file can run under
whichever Translator instance the *including* view already built —
`designer.php`'s, `generate.php`'s, or (if entered through a legacy
standalone entry point that never touched a view) a fallback
`Translator::forView('actions', ...)` that `_dispatcher.php` builds itself.
Since a per-view catalog is invisible to any *other* view's Translator, a
`lang/id/actions.php` file would only resolve correctly some of the time,
depending on entry path — reproducing the exact "wrong scope" class of bug
described above, just via instance mismatch instead of a key-prefix typo.
Putting these keys under `common.php`'s reserved `response` section sidesteps
the problem entirely: every `Translator::forView()` call merges `common.php`
first, so `t('response.document_saved', ...)` resolves identically no matter
which instance is currently active.

## What is NOT translatable

These are **per-document / per-template author content**, not app UI
chrome — never wrap them in `t()`:

- `data-label`, `data-options`, `data-error-msg` attribute values on field /
  TTD / materai placeholders — typed in by hospital staff designing their
  own letter templates.
- Template names, category names.
- The pre-existing `$config->get('designer.page_title', ...)` /
  `$config->get('generate.picker_*', ...)` / `$config->get('generate.error_missing_identity', ...)`
  keys — a separate, older consumer-override mechanism (plain `Config`, no
  `Translator` involved); folding these into the catalog would risk
  breaking existing consumer overrides.
- `{{field_name}}` / `{{tabledb.ns.col}}` document-template mustache
  variables — unrelated syntax, different string universe entirely.
- Internal status/mode discriminator values that get persisted into saved
  template content or config JSON (e.g. a literal `'kosong'`/`'upload'`
  materai-mode value written into a `data-mode="..."` attribute, or the
  `data-nama-field` attribute name itself) — these aren't display text, they're
  part of the on-disk data format for existing templates/documents. Renaming
  them isn't a `t()`-style swap; it needs a backward-compatible dual-read
  migration (see the codebase's existing `logoKiri` → `logo_kiri` pattern for
  precedent), which is a separate, unstarted effort.

AJAX action-endpoint response strings (`actions/**/*.php`, via
`actions/_dispatcher.php`) ARE covered — see "Why `response.*` lives in
`common.php`, not its own catalog" above.

## Reference: Translator API

| Method | Signature | Notes |
|---|---|---|
| `forView()` | `static forView(string $view, string $locale = 'id', ?string $langDir = null): self` | Loads `common.php` + `{view}.php`, merges. |
| `t()` | `t(string $key, array $params = [], ?string $default = null): string` | Never throws. `{param}` interpolation via `strtr()`. |
| `has()` | `has(string $key): bool` | |
| `all()` | `all(): array` | Full merged catalog — used to build the JS dictionary. |
| `getLocale()` | `getLocale(): string` | |
| `getView()` | `getView(): string` | |
